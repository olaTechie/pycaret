# Data Scientist Review — mltoolkit-plugin (2026-04-14)

Reviewer lens: technical rigor — leakage, CV strategy, model-zoo breadth, tuning coverage, interpretability, and reproducible deliverable packaging. All observations cite concrete file:line evidence; nothing is asserted from recall.

## 1. Summary

mltoolkit-plugin delivers a pragmatic sklearn-native alternative to PyCaret for four tasks with a clean stage-based reference pattern and an A/B/C packaging ladder. Technically it is correct in several small but important places: preprocessing is wrapped inside sklearn `Pipeline`/`cross_val_score`, random seeds are consistent, train/test splits are stratified for classification, and SVC enables `probability=True`. However the review surfaces one P0 correctness defect that breaks the documented copy-to-`.mltoolkit/session.py` workflow (sibling module imports and `_PLUGIN_ROOT` computation assume the original location); a P0 false advertisement in README/tune SKILL ("optuna backend") with zero optuna code in the repo; and multiple P1/P2 gaps around tuning breadth (fixed `n_iter=20`, no early-stopping, no budget control), interpretability (no SHAP/PDP/calibration/learning curves), CV strategy (no repeated/group/time-series CV), packaging parity (requirements.txt un-pinned; session.py refit rather than finalize-on-full; LOF serialization silently skipped), and reporting reproducibility (no run manifest / seed / versions pinned into artifacts). Findings are listed in Section 7 as a jsonl block ready for the lead reviewer's synthesis.

## 2. Current-spec gaps

- **Packaged/copied session.py is broken by sibling-module imports.** `skills/classify/references/classify_reference.py:35-42` does `sys.path.insert(0, str(_HERE.parent))` then `import preprocessing` / `import model_zoo`. When the script is copied to `.mltoolkit/session.py` (as the README workflow and every SKILL.md instructs), `_HERE.parent` is `.mltoolkit/`, where those modules do not exist. Same pattern in `regress_reference.py:21-28`, `cluster_reference.py:22-28` (only `model_zoo`), `anomaly_reference.py:20-26`. Tier A (`skills/package/references/tier_a_transform.py:10-18`) simply copies the file — the packaged deliverable inherits the same failure. Tests run the scripts in place (`tests/test_classify.py:41`), so the smoke suite cannot catch this.
- **`from references._shared import ...` likewise assumes `_PLUGIN_ROOT = _HERE.parents[3]`**, which evaluates to a random ancestor when session.py is copied or Tier-A-packaged. See `classify_reference.py:36`, `regress_reference.py:22`, etc.
- **Optuna backend is advertised but not implemented.** `README.md:21` lists optuna as "alternative hyperparameter search backend"; `skills/tune/SKILL.md:24-26` says "If optuna is installed, the generated session can be edited to use optuna instead of RandomizedSearchCV." Grep confirms zero optuna imports, helpers, or code paths in `skills/**` or `references/**`. `references/_shared/deps.py:41-42` has `has_optuna()` but nothing consumes it. Users following the README will not find any optuna code to turn on.
- **Tuning knobs are frozen.** `classify_reference.py:118` and `regress_reference.py:88` hardcode `n_iter=20`, `cv=5`, `scoring="accuracy"` / `"r2"`. No `--n-iter`, `--cv-tune`, `--scoring`, `--early-stopping`, or `--time-budget` CLI flag. SKILL.md iteration prompt "tune more" just says "may require editing the script" (`skills/classify/SKILL.md:58`).
- **RandomizedSearchCV uses fixed `scoring="accuracy"`** in classification regardless of class balance or user intent (`classify_reference.py:135`). The compare stage already switches to `f1_macro`/`roc_auc` by class count (`classify_reference.py:89-94`), but tune ignores that — a minority-positive dataset optimised via compare's `roc_auc` and re-tuned via accuracy will select a different "best".
- **No support for grouped / time-series CV.** `train_test_split` in `classify_reference.py:203` and `regress_reference.py:171` does plain random split; `cross_val_score` uses default `KFold`/`StratifiedKFold`. No `GroupKFold`, `TimeSeriesSplit`, or user-specified `--groups-col` — unsafe for panel / longitudinal data that the README does not warn against.
- **Regression: no stratification on bins for skewed targets.** `regress_reference.py:171` splits without `stratify=pd.qcut(y, k)` or similar; high-skew targets (e.g. income) can leak rare-value tails.
- **Interpretability is permutation-importance-only.** No SHAP, no partial-dependence/ICE, no calibration curve, no learning curve, no decision curves. `classify_reference.py:168` does `permutation_importance` on the final model on the test set. PyCaret ships `interpret_model` (SHAP) + `plot_model(plot='calibration'|'learning'|'vc')` for both classification and regression (see capability reference §1.9, §2.2).
- **No probability calibration hook.** Classification reference never wraps the winning estimator in `CalibratedClassifierCV`; a tuned RF or GBDT emitted from tune is raw. PyCaret ships `calibrate_model` (capability reference §1.6).
- **No class-imbalance remediation.** `skills/classify/SKILL.md:50,59` mentions SMOTE in prose but there is no code path that actually builds an imbalanced-pipeline. Users must hand-edit session.py to wrap in `imblearn.pipeline.Pipeline` — a non-trivial rewrite because the current preprocessor is a sklearn Pipeline/ColumnTransformer with different mro.
- **TargetEncoder usage is permissible but uncontrolled.** `skills/classify/references/preprocessing.py:51-55` instantiates `ce.TargetEncoder()` with defaults — no smoothing, no CV-leave-one-out variant. Because the transformer is inside the Pipeline and inside `cross_val_score`, fit is on the training fold so there is no leakage; but the chosen encoder is not tracked in `schema.csv` or `best_params.json`, so the packaged deliverable silently depends on whether `category_encoders` is installed at run time. Same at `skills/regress/references/preprocessing.py:41-46`.
- **Leaderboard metric columns are task-conditional but README/SKILL do not state which.** `classify_reference.py:89-95` only adds `roc_auc` for binary — multiclass leaderboard has no ranking-style metric. No `logloss`, `MCC`, `Cohen's kappa`, `balanced_accuracy`, `average_precision`, `brier_score`. Regression lacks MAPE, RMSLE, `explained_variance` (capability reference §3.4).
- **Cluster `_prep` silently drops every categorical.** `cluster_reference.py:31-37` calls `df.select_dtypes(include="number")` with no warning to the user when non-numeric columns are present; the SKILL (`skills/cluster/SKILL.md:38-40`) promises a prompt but no code issues one.
- **Cluster refit-for-serialization duplicates work and silently swallows exceptions.** `cluster_reference.py:119-127` creates a fresh estimator, refits, then `try: joblib.dump(...) except Exception: pass`. A DBSCAN failure produces no model file and no stderr — the user has no way to know.
- **Anomaly LOF cannot serialize.** `anomaly_reference.py:101-102` quietly skips `joblib.dump` for LOF (not-novelty). Downstream Tier-B/C packages still advertise "load `model.joblib`" in `tier_b_readme_template.md:34-38` and `tier_c_scaffold/src/predict.py` — the emitted artifact will `FileNotFoundError` on load.
- **No holdout evaluation for cluster or anomaly.** Both fit and score on the full dataset. That is a defensible choice, but no language in the SKILL warns the user that the reported silhouette / anomaly-rate is training-set optimistic.
- **Preprocessing choices are not surfaced anywhere.** The final `ColumnTransformer` configuration (which columns went numeric / low-card OHE / high-card TargetEncoder) is never written to disk; the user cannot audit or reproduce preprocessing decisions without re-running.
- **No reproducibility manifest.** No `environment.json` / `versions.txt` / `sklearn.__version__` / `python_version` written to `.mltoolkit/results/`. `random_state=42` is baked in but not surfaced alongside outputs.
- **`best_params.json` stores `str(v)` not JSON-typed values** (`classify_reference.py:138-139`, `regress_reference.py:104-105`) — downstream consumers cannot round-trip the params into a re-tuned estimator.
- **No `finalize_model` equivalent.** After tune+evaluate, the classification/regression pipeline is refit on `X_train` (`classify_reference.py:225`), not on `X_train ∪ X_test`. PyCaret's `finalize_model` (capability reference §1.11) refits on full data before deployment. The packaged Tier-A/B script therefore deploys a model trained on 80% of the user's data.

## 3. PyCaret-target gaps

Measured against `docs/superpowers/reviews/2026-04-14-pycaret-capability-reference.md`:

- **No SHAP-based interpretability (capability §1.9, §6.2).** `interpret_model(plot='summary'|'reason'|'pdp'|'pfi')` has no mltoolkit analogue.
- **No `calibrate_model` (§1.6).**
- **No fairness auditing (§1.15, §6.1).** PyCaret's `check_fairness(sensitive_features=...)` is absent; no mltoolkit skill accepts a protected attribute column.
- **No `dashboard` / `evaluate_model` interactive wrapper (§1.8, §1.16).** Defensible for a CLI plugin, but the gap should be flagged and a headless HTML equivalent offered (only `reporting.summary_report` exists in `references/_shared/reporting.py`, and no SKILL calls it).
- **Classification model zoo missing canonical IDs (§2.1).** No `qda`, `lda`, `gpc`, `rbfsvm`, `dummy` baseline, `par` (passive-aggressive). `nb` is only `GaussianNB` — no `MultinomialNB` / `BernoulliNB`. `skills/classify/references/model_zoo.py:22-85`.
- **Regression zoo missing (§3.1).** No `lar`, `llar`, `omp`, `br`, `ard`, `par`, `ransac`, `tr` (theil-sen), `huber`, `kr`, `dummy` baseline. `skills/regress/references/model_zoo.py:22-53`.
- **Clustering zoo missing (§4.1).** No `ap`, `meanshift`, `sc` (spectral), `optics`, `birch`, `kmodes`. `skills/cluster/references/model_zoo.py:7-16` ships only 4 of 9.
- **Anomaly zoo missing (§5.1).** No `abod`, `cluster`-LOF, `cof`, `histogram` (HBOS), `knn`, `pca`, `mcd`, `sod`, `sos`. `skills/anomaly/references/model_zoo.py:8-14` ships 4 of ~12. `pca` anomaly detection and `mcd` (Minimum Covariance Determinant) are directly buildable from sklearn (`PCA` + reconstruction error, `MinCovDet`) but absent.
- **No `tune_model` for anomaly / full support for clustering.** Clustering tune is absent in the reference script; anomaly tune likewise (capability §1.4, §4.3).
- **Tuning backends: only one of four canonical backends implemented.** PyCaret exposes `scikit-learn`, `scikit-optimize`, `tune-sklearn`, `optuna` via `search_library` (capability §2.3). mltoolkit ships only `RandomizedSearchCV`; optuna is advertised but absent (§1.3, §2); scikit-optimize and tune-sklearn are not mentioned.
- **Plot catalog much narrower than PyCaret's (§2.2, §3.2).** Missing: `threshold`, `class_report`, `boundary`, `rfe`, `learning`, `manifold`, `calibration`, `vc`, `gain`, `lift`, `ks`, `cooks`, `feature_all`. Clustering missing `silhouette` (as a per-sample plot), `distribution`, `distance`, `tsne`, `umap`. Anomaly missing `tsne`, `umap` (only PCA shipped: `anomaly_reference.py:88-94`).
- **No experiment tracking (§6.3).** PyCaret's `log_experiment='mlflow'` / `'wandb'` / `'comet_ml'` is absent. No MLflow/W&B hook in any reference script.
- **No deployment helpers (§6.4).** No `create_api` / `create_docker` / `create_app` / `convert_model` at skill level — Tier C has a hand-written `api.py` + `Dockerfile` but no Gradio `create_app`, no ONNX/C/Java `convert_model`, and the FastAPI server assumes the model pickle lives next to `api.py` (`tier_c_scaffold/api.py:10`) with no runtime schema validation.
- **No `finalize_model` (§1.11).** Already noted in §2.
- **No `predict_model` with preprocessing-preserving DataFrame output (§1.10).** Tier C's `predict.py:11-13` calls `model.predict(df)` on the raw CSV — acceptable because the pipeline includes preprocessing, but there is no rigour around column-order / schema validation to mirror PyCaret's behaviour.
- **No ensembling (§1.5).** No `ensemble_model` (bagging/boosting wrapper), `blend_models` (soft/hard voting), or `stack_models` (meta-learner). `VotingClassifier`, `VotingRegressor`, and `StackingClassifier`/`StackingRegressor` are sklearn-native and would cost one file each.
- **No `optimize_threshold` (§2.5).** PyCaret ships a threshold sweep for classification; mltoolkit never exposes the decision threshold to the user — every metric is taken at the default 0.5.
- **No `check_drift`.** Out of scope for a plugin review perhaps, but worth flagging.
- **No `get_config` / `set_config` equivalent.** No way to inspect the fitted `ColumnTransformer`, the train/test indices, or the CV splitter (capability §1.17).

## 4. Paper-reporting gaps

Paper-reporting lens: what does a methods section / supplementary appendix need that mltoolkit does not write to disk?

- **No run manifest.** A publishable ML methods section needs: python version, sklearn version, numpy/pandas versions, `random_state`, train/test split proportion, stratification strategy, CV fold count, fold strategy, exact feature list in final order, winning model id + hyperparameters. Of these, mltoolkit writes only `best_params.json` (stringified) and `leaderboard.csv`. Nothing ties a figure to the code version that produced it.
- **No per-fold scores.** `cross_val_score(...).mean()` and `.std()` are stored; individual fold scores are discarded (`classify_reference.py:101-103`). Paper reviewers increasingly ask for fold-by-fold results, and confidence intervals on the mean — one-sigma from 5 folds understates uncertainty.
- **No calibration / decision curves.** Clinical / risk-prediction papers cannot be written with the current output set. Brier score, calibration intercept / slope, and decision curve analysis (DCA) are absent.
- **No subgroup metrics.** Paper tables typically stratify holdout performance by a demographic / clinical variable. No `--subgroup-col` is supported.
- **No confidence intervals around holdout metrics.** `holdout_metrics.json` (`regress_reference.py:118`) stores a point estimate; no bootstrap resampling / DeLong interval for AUC / paired t-test for comparing two models.
- **No comparison-of-models hypothesis test.** Paper reviewers ask whether the top-ranked model is significantly better than the runner-up — `mlxtend.evaluate.paired_ttest_5x2cv` or similar is absent.
- **Figures are saved at 300 DPI PNG+PDF** (`references/_shared/plotting.py:23-42`) — good for manuscript submission — but no `rcParams` for serif fonts or LaTeX integration; no `reporting.df_to_latex` is called from any SKILL despite being defined (`references/_shared/reporting.py:13-15`).
- **No "Table 1" / cohort-characteristics output.** Descriptive tables (mean ± SD for continuous, n (%) for categorical, stratified by outcome or cohort) are standard in clinical ML papers; `setup_reference.py:35-41` only writes a schema.csv.
- **No missing-data handling disclosure.** Preprocessing uses `SimpleImputer(strategy="median"|"most_frequent")` unconditionally — no check for MCAR/MAR/MNAR assumptions, no warning at >5% missingness, no multiple-imputation option (`IterativeImputer`, `sklearn.experimental.enable_iterative_imputer`).
- **No sample-size / power notes.** The skill happily runs on n=50; no floor or warning.
- **No learning curves.** A paper reviewer asking "does adding more data help?" cannot be answered by current outputs.
- **README.md makes workflow claims that are not reflected in tests.** `README.md:40-46` describes copy-to-session.py-and-iterate; the test suite does not simulate that end-to-end path (the P0 sibling-import bug is therefore never hit by CI).

## 5. Prioritized recommendations

### P0

1. **Fix the copy-to-session.py workflow.** Either (a) inline `preprocessing` and `model_zoo` into the reference script so it becomes truly single-file before the copy; (b) have `skills/setup` stage-copy the whole `skills/<task>/references/` directory into `.mltoolkit/` so sibling imports resolve; or (c) replace all sibling imports with an installed-package import path and add the plugin root to `PYTHONPATH` via a wrapper. Add a test that copies the reference to a temp dir and invokes it from there — fail the CI when the import resolution breaks.
2. **Make the optuna claim true or remove it.** Implement a minimal `optuna`-backed path in `tune_model` gated on `deps.has_optuna()` with a `--search-library {random,optuna}` flag, and mirror the SKILL.md language. Or strip the claim from `README.md:21` and `skills/tune/SKILL.md:24-26`.

### P1

3. **Surface tune knobs as CLI args**: `--n-iter`, `--cv-tune`, `--scoring`, `--time-budget`. Default `--scoring` to "same as compare's ranking metric" so the leaderboard winner is also the tune target.
4. **Add `finalize_model` semantics**: after evaluation, refit on `X_train ∪ X_test` and save as `model_finalized.joblib` alongside the current holdout-trained `model.joblib`. Document which artifact downstream `predict.py` consumes.
5. **Add CalibratedClassifierCV hook** behind a `--calibrate {isotonic,sigmoid,none}` flag in the classification reference.
6. **Add SHAP + PDP + learning-curve figures** to `evaluate` (behind `--no-shap` to keep runtime bounded). Gate SHAP on `shap` installed; fall back to permutation importance.
7. **Expand model zoos** to PyCaret parity where sklearn-native: `qda`, `lda`, `dummy`, `par` (classification); `lar`, `llar`, `omp`, `br`, `ard`, `huber`, `ransac`, `dummy` (regression); `optics`, `birch`, `meanshift`, `sc` (cluster); `mcd`, `pca`, `knn`-anomaly (anomaly).
8. **Add per-fold scores to leaderboard** (`leaderboard_folds.csv`) and bootstrap CIs to `holdout_metrics.json`.
9. **Pin requirements.** `skills/package/references/tier_b_transform.py:53-55` emits unpinned deps; capture `importlib.metadata.version()` at session run time and write `requirements.txt` with `==` pins. Same for `tier_c_scaffold/requirements.txt`.
10. **Add a `run_manifest.json`** to `.mltoolkit/results/` with python/sklearn versions, random_state, split strategy, CV folder, feature list, final hyperparameters.
11. **Add `--groups-col` and `--time-col`** to trigger `GroupKFold` / `TimeSeriesSplit` at compare and tune.
12. **Guard the cluster refit-and-serialize path**: log the exception rather than `pass`, and emit `model.joblib` only when the model is serializable (`cluster_reference.py:119-127`).

### P2

13. **Fairness hook**: accept `--sensitive-col` and write a per-group metric table for classification and regression.
14. **Subgroup metrics**: accept `--subgroup-col` and write stratified holdout metrics.
15. **Threshold optimization** for classification: `optimize_threshold(metric='f1'|'mcc'|...)` that writes a new `threshold.json` and re-computes the holdout CM at that threshold.
16. **Ensembling wrappers**: `VotingClassifier` / `StackingClassifier` behind `--stage ensemble`, pulling the top-k from the leaderboard.
17. **Drop-missing / multiple imputation** via `IterativeImputer` behind `--impute {simple,iterative}`.
18. **Headless dashboard**: materialize `reporting.summary_report` as the default last artifact across all four task skills (currently defined but unused).
19. **Tier-B/C README parity**: declare the serialization caveat for LOF and DBSCAN up front; produce a `predict.py` that can fall back to refit-and-predict when the pickle is absent.
20. **CI**: add a test that runs `tier_a_transform` → executes the emitted script against a synthetic dataset, and an end-to-end test that copies session.py into a separate cwd and runs it (closes the P0 gap in coverage).

## 6. Open questions

- Should mltoolkit gain a time-series task (capability reference §7.1 notes `pycaret.time_series` is separate) to reach parity with PyCaret's four-plus-one module surface, or deliberately stay narrower?
- Is the intended user persona a researcher who will edit session.py (current SKILL.md assumes so: "edit session.py to wrap estimator in imblearn.pipeline.Pipeline", `skills/classify/SKILL.md:59`), or a hands-off "one-shot CLI" user? The current SKILL language hedges between the two, which makes it hard to decide whether missing features should be CLI flags (hands-off) or documented hand-edits (researcher).
- What is the expected interaction between `mltoolkit:compare` re-run and the already-written leaderboard? Currently `compare/SKILL.md:24-27` just reruns the stage — no awareness that users may want incremental "add model X to an existing leaderboard" behaviour.
- Should `packaging` guarantee that the Tier-A single file actually runs standalone (no sibling imports, no plugin root)? The current implementation does not, and tests do not verify it.
- How should random_state flow through? One global `42` (current) vs. user-supplied `--seed` propagated to every estimator, `train_test_split`, `RandomizedSearchCV`, and `permutation_importance`.
